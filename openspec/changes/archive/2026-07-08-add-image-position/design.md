## Context

`composables/useEditor.ts` maintains a static `ELEMENTS` registry (`red-bar`, `logo`, `title`, `content`) with shared reactive `positions`/`hidden`/`aspectLocked` state, drag/resize handlers, and CSS-var-based rendering. `layouts/default.vue` restores this state on mount from `data-styles`/`data-hidden`/`data-aspect-locked` attributes baked into the layout file itself by the `/api/save-layout` Vite middleware (`vite.config.ts`) — every slide using a given layout file shares the exact same element positions, because the positions live in the layout, not the slide.

The title element is the one existing case of "extract a DOM node out of normal content flow into its own overlay": the markdown-rendered `<h1>` stays a child of `.content-inner` (wherever the source put it) but is pulled out via `position: absolute` targeting `.slidev-layout.default h1:first-child`, landing in a fixed overlay box positioned independently of its source location.

The paste-to-add-image feature (already shipped) inserts `![](/images/paste-<ts>.png)` inline into a slide's markdown, either via direct textarea splice (editor open) or Slidev's slide-update API (editor closed), riding the existing autosave pipeline. It has no concept of image position today — the image just renders wherever it landed in the markdown, in normal flow.

## Goals / Non-Goals

**Goals:**
- Extract a pasted image out of content flow into its own draggable/resizable/aspect-lockable overlay element, reusing `useEditor.ts`'s existing per-element machinery.
- Offer a below/right preset picker immediately after paste, defaulting to below.
- Make preset selection auto-save without corrupting other slides that share the same layout file.
- Auto-shrink the content box when "right" is chosen, auto-reset it to full width when "below" is chosen (or is the current preset already).

**Non-Goals:**
- Multiple draggable images per slide (v1 is one; see proposal for the "last in document order" selection rule).
- Free-form column splitting beyond the two named presets — no arbitrary N-column layouts.
- Fixing the interaction between narrower content columns and `useAutoFitText`'s sizing (noted as a risk, not solved here).
- Any change to how paste itself detects/uploads images — only what happens after a successful paste.

## Decisions

**Image extraction via CSS selector, same trick as the title.** `layouts/default.vue` targets the last `<img>` inside `.content-inner` (e.g. `.content-inner img:last-of-type`) and pulls it out via `position: absolute`, driven by new `--ed-image-*` CSS custom properties parallel to the existing `--ed-content-*`/`--ed-title-*` vars. No markdown restructuring, no DOM surgery in JS — purely CSS, exactly matching the existing title pattern. "Last in document order" is the selection rule for which image is tracked (see proposal); this is a plain CSS `:last-of-type`, not JS-tracked recency.

**`image` becomes a 5th `ELEMENTS` entry, but conditionally visible.** Unlike the other four (which exist on every slide using the layout), `image` only has meaning when the current slide actually has a trackable `<img>`. `hidden.image` defaults to `true`; the overlay and extraction CSS only apply `v-if`/guarded on an actual image being present. This is a new conditional-existence concept the other four elements don't need — they special-case "hidden" as user-toggled invisibility, not "doesn't exist yet."

**New images default aspect-locked**, using the image's natural pixel dimensions (read client-side via `Image()` before computing preset coordinates). This is an exception to the existing "aspect-lock defaults to off" rule for the other four elements — justified because an arbitrarily-stretched photo looks broken in a way a stretched logo/red-bar usually doesn't.

**Preset selection always saves via the "save as new layout" path, never overwrite.** Since element positions live in the layout file (shared across every slide using it) and image placement is inherently per-slide, overwriting the current layout on preset pick would corrupt every other slide sharing it. Reusing the existing auto-generated `layout-<timestamp>.vue` clone + per-slide frontmatter `layout:` update (already exposed today via the SideEditor's "Save as new layout" checkbox) sidesteps this entirely — each slide that gets a positioned image transparently ends up on its own private layout clone, no user action required. This means the *first* preset pick on a slide forks its layout; a *second* preset change on the same slide (e.g. below → right) overwrites that same already-forked clone (safe, since nothing else uses it) rather than forking again.

**Preset math:**
- *Below*: content box resets to full width (`ELEMENTS.content.initial.w`) at its default x; image is horizontally centered under the content box's current bottom edge, width defaulting to a fraction of slide width, height derived from its aspect ratio.
- *Right*: content box width shrinks to roughly 55–60% of its current width at unchanged x; image occupies the remaining right portion of the content box's original bounds, height matching the content box, width derived from its aspect ratio (clamped to the available space).

**Popover sequencing.** Paste today is fire-and-forget (splice/append, no waiting on render). Showing a popover anchored to the real rendered image requires waiting for Slidev's HMR to re-render the slide with the new `<img>` before the popover can be positioned — a new async step in `global-top.vue`'s paste handler (poll/observe for the image element to appear in the DOM, then anchor).

**Correction found during implementation**: `:last-of-type` is a per-parent-sibling selector, not "last matching element in the whole subtree" — if the tracked image and other images/content sit at different nesting depths (e.g. one `<img>` inside a `<p>`, another inside a `<li>`), a pure `.content-inner img:last-of-type` selector can match more than one element, since each would be the last `img` within its own parent. Pure CSS cannot express "the single last image in document order regardless of nesting," so extraction needs a small amount of JS after all: a `MutationObserver` on `.content-inner` (covering both initial mount and HMR-driven content re-renders) finds all `<img>` elements via `querySelectorAll`, takes the last one in DOM order, and marks it with a `.tracked-image` class (clearing that class from any previously-marked element first). CSS then targets `.content-inner .tracked-image` for the absolute positioning — a class-selector match is unambiguous regardless of nesting.

This observer also *is* the source of truth for `hidden.image`, rather than a separate persisted flag: `hidden.image` is set to `true` whenever no trackable image is found in content, `false` otherwise, purely at runtime. This removes the need for any new persisted "does this layout have an image" marker (e.g. a `data-has-image` attribute) — the existing `positions.image` (`x/y/w/h`) still restores from saved CSS vars via the same generic `VAR_MAP`-driven mechanism the other four elements already use, but if no saved vars exist yet (an image just appeared and was never explicitly positioned/saved), the "Below" preset formula is computed live against the image's natural size and current content box as a sensible default — which is also exactly what the popover's "Below" option would compute and save, so picking it explicitly is a no-op re-save of the same values already showing.

**Second correction found during manual verification**: mirroring `SideEditor.vue`'s "save as new layout" flow turned out to require its `skipHmr` + wait + `window.location.reload()` sequence too, not just the save call — Slidev's layouts virtual module caches the list of known layout names until its own fs watcher notices a new file, and without the reload, the very next slide-info fetch could hit "Unknown layout" for the freshly-forked name. But a full reload wipes any in-memory JS state, which broke the original plan of tracking "has this slide already forked" in an in-memory `Map`. Fixed by not tracking it in memory at all: `saveAsPerSlideLayoutFork` reads the slide's *current* persisted frontmatter `layout:` value and treats anything already matching the auto-generated `layout-<timestamp>` naming convention as an existing fork to update in place (`saveAs: false`) rather than fork again — this is reload-safe since it's persisted state, not memory.

**Third correction found running the full existing test suite**: `image` defaulting to `hidden: true` (the opposite polarity from the other four elements, which default to shown) leaked into the shared `data-hidden` serialization on *every* ordinary layout save, not just image-related ones — since the existing convention only records elements that deviate from a uniform "shown by default" baseline, and `image`'s baseline is inverted, every normal save started including a spurious `"image"` entry (e.g. turning `data-hidden="logo"` into `data-hidden="logo,image"`), breaking an existing exact-match test. Fixed by excluding `image` from that shared list entirely, both when writing it (`/api/save-layout`) and when restoring it (`layouts/default.vue`'s mount-time restoration loop) — its shown/hidden state is already correctly derived at runtime by `updateTrackedImage()` regardless. Accepted trade-off: explicitly hiding a positioned image via its delete button won't survive a page reload (the image reappears, since content still has the underlying `<img>`), mirroring the existing, already-documented limitation that deleting `red-bar`/`logo` is also only undo-able within the same session, not reload-persistent.

**Also found**: `computeBelowPreset` originally just copied the passed-in content width through unchanged, rather than resetting it to the true full-width default — meaning choosing "Below" after a "Right" narrowed the content box would leave it narrow instead of restoring full width. Fixed by threading a `fullContentWidth` parameter (sourced from a new `CONTENT_DEFAULT_WIDTH` export in `useEditor.ts`, so the value isn't duplicated/hardcoded in the preset-math module).

## Risks / Trade-offs

- **[Risk]** Auto-forking a new layout file per positioned image can proliferate `layouts/layout-<ts>.vue` files over time (one per slide that ever got an image positioned) → **Mitigation**: this already happens today for any manual "Save as new layout" use; not a new problem class, just a new trigger for it. Out of scope to solve (e.g. layout cleanup/garbage collection) in this change.
- **[Risk]** Narrower content column (from the "right" preset) can interact with `useAutoFitText`'s font sizing, shrinking text more than expected → **Mitigation**: explicitly a Non-Goal here; flagged for a follow-up change if it proves visually bad in practice.
- **[Risk]** Waiting for the post-paste re-render to anchor the popover adds a new async/observability step to a previously synchronous flow, and could be flaky (slow HMR, race with the autosave throttle) → **Mitigation**: use a bounded `MutationObserver`/poll with a timeout; if the image never appears in time, silently skip the popover rather than hang or error (the image still renders in normal flow either way — the popover is additive UX, not required for correctness).
- **[Trade-off]** "Last in document order" as the single-image selection rule is simple (pure CSS) but can diverge from "most recently pasted" if a user pastes out of order (see proposal) — accepted for v1 simplicity.
